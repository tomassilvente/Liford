Cómo funciona cada API

Una explicación en lenguaje llano de qué hace cada ruta, qué recibe y qué devuelve — sin asumir que sabés cómo funciona.

Primero: ¿qué es una API route?

Cada vez que hacés algo en la app (guardar un gasto, crear una sesión, borrar un cliente), el browser le manda un pedido al servidor. Ese pedido llega a una API route — una función que vive en el servidor, hace lo que tiene que hacer, y le responde al browser.

Hay cuatro tipos de pedido, y cada uno tiene una intención:

G
GET — "Dame información"
No cambia nada. Solo lee la DB y devuelve datos. Ej: traer la lista de gastos del mes.
P
POST — "Creá algo nuevo"
Recibe datos y crea un nuevo registro en la DB. Ej: crear un gasto, una billetera, una sesión.
A
PATCH — "Modificá algo que ya existe"
Recibe los campos a cambiar y actualiza el registro. Ej: editar el nombre de una billetera, cambiar el estado de una sesión.
D
DELETE — "Eliminá algo"
Borra un registro de la DB. Ej: eliminar un gasto, un cliente, una inversión.

El patrón que siguen TODAS las rutas

Sin excepción, cada ruta hace estos pasos en orden:

Patrón universal
1
Verificar que el usuario esté logueado
Llama a getApiSession(). Lee la cookie de sesión y verifica el JWT. Si no hay sesión válida, devuelve error 401 y para todo.
↓
2
Leer lo que mandó el browser
Para GET: lee los parámetros de la URL (ej: ?mes=2026-04). Para POST/PATCH: lee el JSON del body (request.json()).
↓
3
Verificar que los datos le pertenecen al usuario
Siempre filtra por userId. Esto evita que alguien acceda a los datos de otro usuario aunque consiga el ID.
↓
4
Ejecutar la operación en la base de datos
Acá es donde realmente pasa algo: create, update, delete, findMany. Usando Prisma.
↓
5
Responder al browser
Response.json(datos) devuelve el resultado. El browser lo recibe y actualiza la pantalla.

Billeteras

Las billeteras son las cuentas donde guardás plata. Pueden ser ARS o USD. El balance se actualiza automáticamente cada vez que registrás un gasto o ingreso vinculado a esa billetera.

GET /api/finanzas/wallets Traer todas las billeteras

Busca en la DB todas las billeteras que le pertenecen al usuario logueado y las devuelve en orden por nombre.

Recibe
Nada. Solo la cookie de sesión para saber quién es.
Devuelve
Array de billeteras: [ { id, name, currency, balance }, ... ]

¿Cuándo se llama? Al abrir /finanzas/billeteras y también en el layout al cargar el QuickAdd (necesita saber las billeteras disponibles).

POST /api/finanzas/wallets Crear billetera

Crea una nueva billetera con saldo inicial en cero.

Recibe
{ name: "Mercado Pago", currency: "ARS" }
Devuelve
La billetera recién creada con su id generado.
PATCH /api/finanzas/wallets/[id] Editar billetera

Actualiza el nombre y/o el balance de una billetera. Si mandás solo el nombre, solo cambia el nombre. Si mandás solo el balance, solo cambia el balance. Podés mandar ambos.

Recibe
{ name?: "Nuevo nombre", balance?: 50000 }

Todos los campos son opcionales.
Devuelve
La billetera actualizada.
Importante: cuando editás el balance directamente (tocando el número en la pantalla), estás usando esta ruta. El balance nuevo reemplaza al anterior — no suma ni resta, sobreescribe. Esto es intencional para correcciones manuales.
DELETE /api/finanzas/wallets/[id] Eliminar billetera

Elimina la billetera. Las transacciones que estaban vinculadas a ella quedan en la DB pero sin referencia a ninguna billetera.

Recibe
Solo el id en la URL.
Devuelve
Nada (status 204 = "ok, sin contenido").

Cuentas Foráneas

Cuentas en el exterior (Wise, Payoneer, brokers, etc.). A diferencia de las billeteras normales, el balance se puede editar manualmente porque son cuentas externas que no se actualizan automáticamente con transacciones.

GET/api/finanzas/foreign-accountsTraer cuentas foráneas

Igual que wallets. Devuelve todas las cuentas foráneas del usuario.

POST/api/finanzas/foreign-accountsCrear cuenta foránea
Recibe
{ name: "Wise", currency: "USD", balance: 0, notes?: "..." }
Devuelve
La cuenta creada.
PATCH/api/finanzas/foreign-accounts/[id]Editar cuenta foránea

Permite cambiar el nombre, el balance o las notas. El balance se puede sobreescribir directamente para sincronizarlo con el balance real de la cuenta.

Recibe
{ name?, balance?, notes? }
Devuelve
La cuenta actualizada.
DELETE/api/finanzas/foreign-accounts/[id]Eliminar cuenta foránea

Elimina la cuenta foránea.

Gastos

POST/api/finanzas/gastosRegistrar un gasto

Esta es la ruta más importante del módulo de finanzas. Hace dos cosas al mismo tiempo (atómicamente, usando una transacción de Prisma — si una falla, la otra no se ejecuta).

Recibe
{ description, amount, category, date, walletId? }

O en lugar de walletId: foreignAccountId
Devuelve
La transacción creada con su id.
Lo que pasa adentro
1
Verifica que la billetera existe y es tuya
Busca la wallet por id Y por userId. Si alguien intenta usar el id de la billetera de otro usuario, falla.
↓
2
db.$transaction([...]) — las dos cosas juntas
Crea la Transaction en la tabla de transacciones. Y simultáneamente hace balance: { decrement: amount } en la wallet. Las dos operaciones se ejecutan o ninguna.

Ingresos

POST/api/finanzas/ingresosRegistrar un ingreso

Exactamente igual que gastos pero al revés. En lugar de decrement, hace increment en el balance de la wallet. La transacción queda con type: INCOME y source: PERSONAL.

Recibe
{ description, amount, category, date, walletId? }
Devuelve
La transacción creada.

Transacciones (editar y borrar)

POST/api/finanzas/transactionsParsear con IA (QuickAdd)

Esta ruta es la que usa el QuickAdd cuando escribís en lenguaje natural. No guarda nada. Solo parsea el texto y devuelve lo que entendió para que vos lo confirmes.

Recibe
{ text: "gasté 5000 en el super" }
Devuelve
{ type: "EXPENSE", amount: 5000, currency: "ARS", description: "Supermercado", category: "Alimentación" }
Claude Haiku: internamente llama a expense-parser.ts que llama a Claude Haiku con un prompt específico. Haiku es más rápido y barato que Sonnet para esta tarea porque la respuesta esperada es simple y estructurada.
PATCH/api/finanzas/transactions/[id]Editar transacción

Permite cambiar la descripción, categoría y monto de una transacción ya guardada. Se usa cuando clickeás el ícono de editar en una fila de gasto/ingreso.

Recibe
{ description?, category?, amount? }
Devuelve
La transacción actualizada.
No actualiza el balance de la wallet. Si cambiás el monto de un gasto de $1000 a $2000, el balance de la wallet no cambia automáticamente. Esto es una limitación conocida — para ajustes de monto, lo más limpio es eliminar y volver a registrar.
DELETE/api/finanzas/transactions/[id]Eliminar transacción

Elimina una transacción. Tampoco revierte el balance de la wallet — la transacción desaparece del historial pero el dinero no "vuelve" a la billetera automáticamente.

¿Por qué no revierte el balance? Porque podría romper casos donde el usuario ajustó el balance manualmente. Si necesitás que el balance refleje la eliminación, editá el balance de la billetera directamente tocando el número.

Inversiones

Acciones y criptomonedas. El precio actual no se guarda en la DB — se consulta en Yahoo Finance cada vez que abrís la página.

GET/api/finanzas/inversionesListar inversiones

Devuelve todas las posiciones del usuario. No incluye el precio actual — ese se pide por separado a Yahoo Finance desde la página.

POST/api/finanzas/inversionesAgregar activo

Agrega un activo a la cartera. Si el ticker ya existe, recalcula el precio promedio ponderado en vez de crear un duplicado.

Recibe
{ ticker: "GGAL", type: "STOCK", quantity: 100, avgBuyPrice: 5.20 }
Devuelve
La inversión creada o actualizada.
Precio promedio ponderado: si tenías 100 acciones a $5 y comprás 50 más a $6, el nuevo promedio es (100×5 + 50×6) / 150 = $5.33.
PATCH/api/finanzas/inversiones/[id]Editar posición

Permite corregir la cantidad o el precio promedio de una posición. Útil si te equivocaste al cargar.

Recibe
{ quantity?, avgBuyPrice? }
Devuelve
La inversión actualizada.
GET/api/finanzas/cotizaciones/[ticker]Precio histórico de un activo

Esta ruta actúa como intermediario entre tu app y Yahoo Finance. El browser le pide a tu servidor, tu servidor le pide a Yahoo, y le devuelve el resultado al browser. Esto evita problemas de CORS (Yahoo no deja que el browser le pida directamente).

Recibe
[ticker] en la URL + ?range=1mo (opciones: 1w, 1mo, 3mo, 6mo, 1y)
Devuelve
{ ticker, range, data: [{date, price}] } — array de precios históricos para el gráfico.
Cache de 5 minutos: Next.js cachea automáticamente la respuesta 5 minutos (next: { revalidate: 300 }). Si dos usuarios piden el mismo ticker en menos de 5 minutos, Yahoo solo recibe un pedido.

Gastos / Ingresos Recurrentes

Cosas que pasan todos los meses el mismo día: Netflix, sueldo, alquiler. Se aplican automáticamente cuando abrís finanzas.

POST/api/finanzas/recurrentesCrear recurrente
Recibe
{ description: "Netflix", amount: 8000, currency: "ARS", category: "Suscripciones", dayOfMonth: 15, transactionType: "EXPENSE", walletId: "..." }
Devuelve
El recurrente creado con isActive: true y lastApplied: null.
PATCH/api/finanzas/recurrentes/[id]Editar / activar / desactivar

Sirve para editar cualquier campo del recurrente, incluyendo isActive para pausarlo sin borrarlo.

¿Cuándo se aplican automáticamente? Cada vez que abrís el módulo de finanzas, el layout llama a applyRecurringExpenses(). Esta función busca todos los recurrentes activos cuyo dayOfMonth ya pasó en el mes actual y cuyo lastApplied es del mes anterior. Los aplica todos juntos y actualiza el balance.

Importar y Exportar

POST/api/finanzas/importParsear archivo (paso 1 de 2)

Recibe el archivo, lo parsea y devuelve una preview sin guardar nada. Es el paso de "¿esto es lo que querés importar?" antes de confirmar.

Recibe
Archivo multipart: .xlsx, .json o .mmbackup
Devuelve
Array de transacciones parseadas: [{ date, type, amount, currency, category, description }]
Tres parsers distintos según el formato. El más complejo es el .mmbackup (exportación de la app Money Manager): es un ZIP con 8 bytes propietarios, contiene una DB SQLite interna con las transacciones.
POST/api/finanzas/import/confirmConfirmar importación (paso 2 de 2)

Recibe el array de transacciones que se revisó en el paso anterior y las inserta todas en la DB de una sola vez.

Recibe
El mismo array del paso anterior: { transactions: [...] }
Devuelve
{ imported: 47 } — cuántas se importaron.
Usa createMany(): inserta todas las transacciones en un solo query de DB, mucho más eficiente que hacer un INSERT por cada una. No actualiza balances de billeteras — la importación es solo para historial.
GET/api/finanzas/exportExportar a Excel

Trae todas las transacciones del usuario de la DB, las convierte a formato Excel con la librería xlsx y las devuelve como descarga directa.

Recibe
Nada. Solo la sesión.
Devuelve
Archivo .xlsx con columnas: Fecha, Tipo, Descripción, Categoría, Monto, Moneda.

Fotografía — Clientes

Un cliente es un equipo, persona o empresa a quien le sacás fotos. Cada sesión está vinculada a un cliente.

GET/api/fotografia/clientsListar clientes

Devuelve todos los clientes del usuario, ordenados por nombre. Incluye el conteo de sesiones de cada uno (_count.sessions) para mostrar "3 sesiones" en la lista.

POST/api/fotografia/clientsCrear cliente
Recibe
{ name: "Racing Club", instagram?: "racing", phone?: "+54...", notes?: "..." }
Devuelve
El cliente creado.
PATCH/api/fotografia/clients/[id]Editar cliente

Antes de actualizar, verifica que el cliente le pertenezca al usuario. Esto es importante porque alguien podría intentar editar el cliente de otro usuario si consigue el ID.

Recibe
{ name?, instagram?, phone?, notes? }
Devuelve
El cliente actualizado.
DELETE/api/fotografia/clients/[id]Eliminar cliente

Elimina el cliente. Si tenía sesiones, Prisma las elimina en cascada (depende de cómo está configurado el schema).

Fotografía — Sesiones

Una sesión es un trabajo fotográfico: un partido, un evento, un media day. Es el modelo central del módulo.

GET/api/fotografia/sessionsListar sesiones

Devuelve todas las sesiones del usuario con el nombre del cliente incluido. Acepta filtro opcional por estado.

Recibe
?status=PENDING (opcional). Sin el param, devuelve todas.
Devuelve
Array de sesiones con client: { name, instagram } incluido.
POST/api/fotografia/sessionsCrear sesión

La ruta más compleja de fotografía. Hace 3 cosas en secuencia:

1
Verifica el cliente
Busca el cliente por clientId y verifica que le pertenezca al usuario. Si no existe, 404.
↓
2
Crea la sesión en la DB
Estado inicial: PENDING. La fecha viene con offset -03:00 del form para guardarse correctamente en UTC.
↓
3
Promise.all(): Drive + Calendar en paralelo
Llama a createSessionFolder() y createCalendarEvent() al mismo tiempo. Ambas pueden fallar sin romper la creación de la sesión. Si tienen éxito, guarda la driveUrl y el googleCalendarId en la sesión.
Recibe
{ clientId, type, date, price, currency, durationMinutes?, eventName?, notes? }
Devuelve
La sesión creada con driveUrl y googleCalendarId si se pudieron crear.
PATCH/api/fotografia/sessions/[id]Editar sesión (la más importante)

Actualiza cualquier campo de la sesión. Pero además tiene lógica especial que se activa según qué cambió:

★
Si el estado cambia a PAID (y no tenía transacciones aún)
Crea automáticamente un ingreso en Finanzas: type=INCOME, source=PHOTOGRAPHY, sessionId=[id]. Esto es el puente entre fotografía y finanzas.
★
Si cambió la fecha o la duración, y hay un evento en Calendar
Llama a updateCalendarEvent() para sincronizar el evento con la nueva fecha/hora.
Recibe
Cualquier campo de la sesión: { status?, date?, price?, driveUrl?, photosDelivered?, notes?, ... }
Devuelve
La sesión actualizada con los datos del cliente incluidos.
DELETE/api/fotografia/sessions/[id]Eliminar sesión

Hace tres cosas en orden:

1
Elimina las transacciones de finanzas vinculadas
db.transaction.deleteMany({ where: { sessionId: id } }) — elimina el ingreso que se había creado en finanzas cuando se marcó PAID.
↓
2
Elimina la sesión
db.session.delete()
↓
3
Elimina el evento de Google Calendar
Si la sesión tenía googleCalendarId, llama a deleteCalendarEvent(). Si falla, no importa — la sesión ya está eliminada.

Fotografía — Drive

GET/api/fotografia/drive/[folderId]Listar fotos de una carpeta

Devuelve solo las imágenes (no carpetas) de una carpeta específica de Drive. Se usa en la página de detalle de sesión para mostrar las fotos entregadas.

Recibe
[folderId] en la URL — el ID de la carpeta de Drive.
Devuelve
Array de fotos: [{ id, name, thumbnailLink, webViewLink, mimeType }]
GET/api/fotografia/drive/browse/[folderId]Navegar Drive completo

Similar al anterior pero devuelve carpetas E imágenes mezcladas. Es el que usa el explorador de Drive en el Editor de Stories. Mandando root como folderId mostrás la raíz de Mi unidad.

Las carpetas van primero (orderBy: "folder,name"), así podés navegar la estructura antes de ver las imágenes.
GET/api/fotografia/drive/thumbnail/[fileId]Proxy de thumbnail

Esta ruta existe porque los thumbnails de Drive requieren autenticación con Google. El browser por sí solo no puede pedirlos. Entonces el browser le pide a tu servidor, tu servidor le pide a Google con el token de OAuth, y le devuelve la imagen al browser.

1
Browser pide /api/fotografia/drive/thumbnail/[fileId]
↓
2
Servidor obtiene el thumbnailLink del archivo via Drive API
La URL del thumbnail de Google viene con autenticación. El servidor la llama con el access token de OAuth.
↓
3
Devuelve la imagen al browser con Cache-Control de 1 hora
El browser la cachea, por lo que la siguiente vez que la pida no necesita ir al servidor.

Fotografía — Editor de Stories con IA

POST/api/fotografia/editorGenerar Story con Claude
Recibe
{ prompt: "...", photoUrl?: "...", previousHtml?: "..." }

photoUrl: si el usuario eligió una foto.
previousHtml: si está usando "Ajustar diseño actual".
Devuelve
{ html: "..." }

HTML completo y autocontenido de 1080×1920px.
1
Construye el mensaje para Claude
Si hay previousHtml, el mensaje dice "partiendo de este diseño, aplicá estos cambios". Si hay photoUrl, dice "usá esta foto de fondo".
↓
2
Llama a Claude Sonnet 4.6
El system prompt define el estilo de @stomas.ph: Bebas Neue, colores contextuales por deporte, overlay oscuro en fotos, logo al pie. Claude devuelve HTML puro.
↓
3
extractHtml() limpia la respuesta
A veces Claude envuelve el HTML en bloques de código markdown (```html ... ```). Esta función extrae el HTML limpio en cualquier caso.
↓
4
Devuelve { html } al browser
El componente DesignEditor.tsx lo muestra en el iframe del preview escalado.

WhatsApp Bot

GET/api/whatsapp/webhookVerificación de Meta

Meta llama a esta ruta cuando configurás el webhook por primera vez para verificar que el servidor es tuyo. Responde el hub.challenge si el token coincide con WHATSAPP_VERIFY_TOKEN en el .env.

POST/api/whatsapp/webhookProcesar mensaje entrante

Meta llama a esta ruta cada vez que te llega un mensaje. El flujo completo:

1
Verifica que el número es el autorizado
Si viene de otro número, lo ignora silenciosamente (devuelve 200 para que Meta no reintente).
↓
2
Procesa el tipo de mensaje
Si es texto: usa el texto directo. Si es audio: lo transcribe via Groq Whisper. Si es otro tipo (foto, sticker, etc.): responde que no lo soporta.
↓
3
parseTransaction() con Claude Haiku
Detecta tipo (gasto/ingreso), monto, moneda, descripción y categoría del texto libre.
↓
4a
Si categoría = "Fotografía" + tipo = INCOME
Busca o crea el cliente, crea la sesión como PAID, crea la transacción con source=PHOTOGRAPHY. El ingreso queda tanto en fotografía como en finanzas.
4b
Si es cualquier otro gasto/ingreso
Crea la transacción normal con source=PERSONAL y actualiza el balance de la wallet.
↓
5
Responde por WhatsApp
Manda un mensaje de confirmación con el resumen de lo que se registró.

Autenticación

POST/api/auth/loginIniciar sesión
Recibe
{ username, password }
Devuelve
Si es correcto: setea la cookie y devuelve { ok: true }. Si no: error 401.

bcryptjs compara la contraseña ingresada con el hash guardado en la DB. Si coincide, crea el JWT y lo guarda en la cookie liford_session.

POST/api/auth/logoutCerrar sesión

Elimina la cookie de sesión. El browser ya no puede autenticarse hasta que haga login de nuevo.

GET/api/auth/me¿Quién soy?

Devuelve los datos del usuario logueado (sin la contraseña). Útil para saber en el browser qué usuario está activo.

POST/api/auth/passwordCambiar contraseña
Recibe
{ currentPassword, newPassword }
Devuelve
{ ok: true } o error si la contraseña actual es incorrecta.

Verifica primero que la contraseña actual sea correcta antes de cambiarla. El nuevo hash se guarda en la DB.

Resumen del patrón

Todas las rutas siguen el mismo camino: verificar sesión → leer datos → verificar ownership → operar en DB → responder. La diferencia entre rutas es qué operación hacen en el paso de DB y qué lógica adicional disparan (Google Calendar, Drive, Claude, etc.).

Si en algún momento algo falla, la respuesta es siempre un JSON con { error: "..." } y un código HTTP apropiado (401 si no autenticado, 404 si no existe, 400 si faltan datos, 500 si hay un error del servidor).
